# CANAnalyzerPro 架构设计文档

## 1. 设计目标（v3）

本版本架构目标为：在高速 CAN/CAN-FD 数据流下实现长期稳定运行，且在任何场景下都不让采集链路被下游反压。

核心原则：
- 采集链路优先级最高，任何模块都不得阻塞采集线程
- 控制面与数据面彻底分离
- 先保真存储，后展示与分析
- 任一消费者异常不影响其他消费者
- 可观测、可恢复、可回放

---

## 2. 总体架构（不背压采集）

```
┌──────────────────────────────────────────────────────────────────┐
│                           UI进程 (can-ui)                        │
│  MainWindow / CANView / Playback / Sender / Stats / Settings     │
└──────────────────────────────────────────────────────────────────┘
                              ↑ 订阅流 / 查询
                              │
┌──────────────────────────────────────────────────────────────────┐
│                     服务层进程 (can-service)                     │
│  SubscriptionManager / QueryEngine / ExportWorker / Replay       │
│  - 多消费者隔离                                                    │
│  - 导出与统计异步执行                                               │
└──────────────────────────────────────────────────────────────────┘
                              ↑ 无锁读
                              │
┌──────────────────────────────────────────────────────────────────┐
│                     存储层进程 (can-store)                       │
│  HotRing (内存) + WAL (顺序日志) + Index (查询索引)              │
│  - 写入优先                                                        │
│  - 崩溃恢复                                                        │
└──────────────────────────────────────────────────────────────────┘
                              ↑ 单向写入（不可反压）
                              │
┌──────────────────────────────────────────────────────────────────┐
│                     核心采集进程 (can-core)                      │
│  DeviceActor / ProtocolDecoder / FramePool / IngestPipeline      │
│  - 多设备隔离                                                      │
│  - 零拷贝批处理                                                    │
│  - 永不被UI/导出/分析反压                                           │
└──────────────────────────────────────────────────────────────────┘
```

## 3. 控制面与数据面

### 3.1 控制面（低频）
- 设备连接/断开
- 通道模式切换（RX/TX/NORMAL）
- 参数配置（波特率、协议模式、过滤规则）
- 回放任务控制、导出任务控制

### 3.2 数据面（高频）
- 原始字节流接收
- 协议解码（GVRET/ZCAN/后续协议）
- 帧标准化（统一 FrameRecord）
- 批量写入 HotRing + WAL
- 发布给订阅者（只读流）

控制面与数据面之间仅通过轻量命令通道交互，不共享高频对象。

---

## 4. 关键模块设计

### 4.1 DeviceActor（每设备独立）
- 每个设备一个独立 Actor（线程或事件循环）
- 连接状态机、重连策略、心跳监测
- 单设备故障隔离，不影响其他设备

### 4.2 ProtocolDecoder（插件化）
- 统一解码接口：输入字节，输出标准帧批次
- 版本能力协商（例如 ZCAN 扩展能力）
- 解析异常自愈：局部重置状态机，不中断主循环

### 4.3 FramePool（对象池）
- 固定大小块预分配
- 句柄化传递，减少拷贝与分配抖动
- 峰值阶段避免频繁 malloc/free

### 4.4 IngestPipeline（采集流水线）
- 批量解码
- 批量提交存储
- 严格单向：采集 -> 存储，不接受下游“慢”的反馈

### 4.5 StorageEngine（HotRing + WAL + Index）
- HotRing：秒级到分钟级实时窗口
- WAL：顺序追加，保证数据不丢
- Index：面向查询与导出的时间/设备/通道/ID 索引

### 4.6 SubscriptionManager（多消费者隔离）
- 每个消费者拥有独立游标与队列
- 慢消费者只影响自己
- UI 仅订阅展示级流，导出与分析走独立消费组

### 4.7 ExportWorker（完全异步）
- 导出任务进入后台队列
- 支持快照导出（一致性视图）
- UI 只收进度与结果，不执行重 IO

## 5. 数据流（不反压版本）

### 5.1 接收与落盘主链路
```
DeviceActor接收字节流
  -> ProtocolDecoder批量解码
  -> FramePool分配帧块
  -> IngestPipeline批量提交
  -> StorageEngine写HotRing + WAL
  -> 发布只读流给各消费者
```

### 5.2 UI 消费链路
```
UI订阅设备/通道/过滤条件
  -> SubscriptionManager分配独立游标
  -> 按刷新节奏拉取批次
  -> 局部渲染与统计更新
```

### 5.3 导出链路
```
用户发起导出
  -> ExportWorker创建快照点
  -> 按索引扫描与格式化
  -> 后台写文件
  -> 回传进度与完成事件
```

## 6. 性能策略（极致吞吐）

### 6.1 绝对优先级
- 采集线程 > 存储线程 > 路由线程 > UI线程
- 采集线程不执行任何阻塞 IO（除必要驱动读取）

### 6.2 零拷贝与批处理
- 帧跨模块传递使用句柄/引用
- 批量处理默认启用，避免逐帧信号风暴
- 批次大小自适应（依据 CPU、队列深度、抖动）

### 6.3 显示降级而非采集降级
- 过载时优先降低 UI 刷新频率
- 允许 UI 抽样显示，但原始数据照常存储
- 导出永远从存储层读取完整数据

### 6.4 预算化容量管理
- 以“内存预算”替代“固定帧数阈值”
- 以“磁盘写入预算”替代“临时阻塞策略”
- 队列与缓冲区参数可运行期调优

---

## 7. 稳定性策略（长期运行）

### 7.1 故障隔离
- 多进程边界隔离 UI 与核心采集
- 单设备 Actor 崩溃可独立恢复
- 消费者异常自动摘除，不影响主链路

### 7.2 崩溃恢复
- WAL + checkpoint
- 启动时从最近 checkpoint 快速恢复
- 恢复后继续追加，不破坏历史数据

### 7.3 一致性与可追溯
- 每帧生成全局序号与时间戳
- 支持按偏移回放、按范围重放
- 导出与分析可复现同一数据集

### 7.4 可观测性
- 实时指标：输入帧率、写入延迟、丢帧率、队列深度、UI延迟
- 健康状态：设备在线率、重连次数、导出失败率
- 告警等级：提示/警告/严重

---

## 8. SLO（服务级目标）

建议初始目标：
- 吞吐：持续 50k 帧/秒，峰值 100k 帧/秒（按设备组合评估）
- 延迟：采集到可见 P99 < 50ms
- 丢帧：正常工况 0 丢帧；极端工况仅允许显示侧降级
- 稳定：7x24 小时持续运行，无不可恢复状态
- 恢复：核心进程异常后 10 秒内自动恢复采集能力

---

## 9. 参数建议（首版）

建议初始参数（可按机器配置调整）：
- Ingest 批次：256~1024 帧/批
- HotRing：按 30~120 秒窗口配置
- WAL 段文件：128MB~512MB/段
- WAL 刷盘：10~50ms 周期 + 关键事件立即刷盘
- UI 刷新：50~100ms 定时批量刷新
- 导出并发：1~2 个后台任务，避免争用磁盘带宽

## 10. 迁移路线图（从现有实现平滑升级）

### Phase 1：先观测后重构（1~2周）
- 统一关键指标采集
- 补齐端到端性能基线
- 明确瓶颈占比

### Phase 2：下沉重任务（2~4周）
- 导出完全异步化
- 统计与查询从 UI 剥离
- 路由链路统一批处理

### Phase 3：存储优先（3~6周）
- 引入 HotRing + WAL
- 支持快照导出与偏移回放
- 建立恢复流程

### Phase 4：进程拆分（4~8周）
- 拆出 can-core / can-store / can-service
- 完成 IPC 协议与订阅模型
- 压测、故障注入、回归自动化

---

## 11. 架构约束（v3）

强制约束：
- 采集主链路不得依赖 UI 生命周期
- 采集线程禁止直接进行文件格式化导出
- 任何消费者不得对采集线程施加阻塞
- 导出必须基于存储快照，不直接扫 UI 模型
- 新功能必须定义指标与失败恢复路径

推荐约束：
- 关键模块保持可替换（解码器、存储后端、导出器）
- 所有跨模块协议版本化
- 所有运行时参数支持动态下发和热更新
